## 1. 目的

本書は、AI自動レビュー・ユニットテスト生成システムの技術的な実装方針・構成・インタフェース仕様を定義する。

## 2. システム全体構成

### 2.1 コンポーネント一覧

| コンポーネント | 役割 | 備考 |
|---|---|---|
| GitLab | MR管理・Webhook発火・コメントUI | SaaS / セルフホスト問わず |
| Jenkins | ビルド・テスト実行・CI基盤 | Pipeline（Jenkinsfile）で管理 |
| Webhook受信サーバ | GitLabイベントを受信し処理を振り分け | Node.js / Python 等で実装 |
| CLIエージェント | レビュー・修正・ユニットテスト生成を実行 | Claude Code（Claude系） |
| 指摘管理DB | 指摘IDと状態を永続化 | SQLite / PostgreSQL |
| Git操作環境 | AIによるブランチへのコミット | gitコマンド / API |

### 2.2 最小構成（1台構成）

```
[GitLab] ──Webhook──▶ [Webhook受信サーバ]
                              │
                    ┌─────────┴─────────┐
                    ▼                   ▼
               [Jenkins]          [CLIエージェント]
                    │                   │
                    └─────────┬─────────┘
                              ▼
                       [指摘管理DB]
                              │
                              ▼
                       [Git操作環境]
```

## 3. データモデル

### 3.1 指摘（Finding）

```sql
CREATE TABLE findings (
    id          TEXT PRIMARY KEY,  -- AI指摘: R1, R2, ...  / 人間指摘: H1, H2, ...
    mr_id       TEXT NOT NULL,
    source      TEXT NOT NULL,     -- AI / HUMAN
    status      TEXT NOT NULL,     -- OPEN / PENDING_APPROVAL / APPLIED / RESOLVED / REJECTED
    author      TEXT,              -- sourceがHUMANの場合はGitLabユーザー名
    file_path   TEXT,
    line_start  INTEGER,
    line_end    INTEGER,
    description TEXT,
    suggestion  TEXT,              -- before/after形式の修正提案（任意）
    created_at  DATETIME,
    updated_at  DATETIME
);
```

### 3.2 状態遷移

GitLab DiscussionsのClose（Resolve）状態はDBで管理しない。  
全指摘のClose状態は `blocking_discussions_resolved` イベント発火時にDiscussions APIで確認する。

```
OPEN
 ├─▶ PENDING_APPROVAL（/ai apply 実行 → 修正案提示）
 │     └─▶ APPLIED（/ai approve 実行 → ブランチへコミット）
 ├─▶ RESOLVED（/ai resolve 実行 → 手動修正扱い）
 └─▶ REJECTED（/ai reject 実行）
```

> **全指摘Close後のフロー：**  
> GitLab Discussionsが全Resolve（全指摘Close）された時点で再レビューを自動起動し、新たな問題がなければユニットテスト生成フェーズへ進む。

## 4. Webhook受信サーバ

### 4.1 受信イベント一覧

| GitLabイベント | トリガー条件 | 処理 |
|---|---|---|
| `merge_request` / `opened` | MR作成 | Jenkinsビルドジョブ起動 |
| `note` / MRコメント | `/review <内容>` | 人間指摘をOPEN状態でDB保存・MRにコメント投稿 |
| `note` / MRコメント | `/ai apply <ID>` | Discussions APIでスレッドResolve確認 → OK時のみ修正案生成・MRコメント投稿 |
| `note` / MRコメント | `/ai approve <ID>` | ブランチへ修正コミット・再評価 |
| `note` / MRコメント | `/ai reject <ID>` | 指摘をREJECTEDに遷移 |
| `note` / MRコメント | `/ai test` | ユニットテスト生成を手動トリガー |
| `merge_request` / `update` | `blocking_discussions_resolved: true` | Discussions APIで全Resolve確認 → 再レビュー実行 → ユニットテスト生成フェーズへ移行 |

> **⚠️ GitLab制約：個別コメントのResolveはWebhookで検知できない。**  
> 全指摘のClose（全Resolve）は `blocking_discussions_resolved` イベントで検知し、再レビューをトリガーする。

### 4.2 コマンドパーサー仕様

```
受信コメント本文を正規表現で解析
  /review (.+)                               # 人間指摘の登録
  /ai (apply|approve|reject|test)(\s+\w+)?   # AI操作コマンド
```

#### `/review` コメント仕様

```
/review <指摘内容>
[file: <ファイルパス>]     # 任意
[line: <行番号>]           # 任意
```

例：
```
/review 変数名が仕様と一致していない
file: src/order.py
line: 88
```

- 登録時に `H1`, `H2`, ... 形式のIDを採番
- 登録完了後、MRにコメントでID・内容を返信
- 登録者のGitLabユーザー名を `author` として記録

- コマンド不正・対象IDが存在しない場合はMRにエラーコメントを返す
- 操作権限は GitLab ロール（Maintainer / Developer）で制御

### 4.3 Webhook署名検証

- GitLab Secret Token をHTTPヘッダ `X-Gitlab-Token` で検証
- 不一致時は 401 を返し処理しない

## 5. Jenkinsパイプライン設計

### 5.1 ビルドパイプライン（MR作成時）

```groovy
pipeline {
    agent any
    stages {
        stage('Checkout') { ... }
        stage('Build & Lint') {
            steps {
                // コンパイル・Lint実行
                // 失敗時はパイプライン終了（AIレビューをスキップ）
            }
        }
        stage('AI Review') {
            steps {
                // CLIエージェント呼び出し
                // 指摘をDBに保存・GitLabにコメント投稿
            }
        }
    }
}
```

### 5.2 テスト実行パイプライン（ユニットテスト生成後）

```groovy
pipeline {
    agent any
    stages {
        stage('Checkout') { ... }
        stage('Existing Tests') {
            steps {
                // 既存テスト実行・カバレッジ取得
            }
        }
        stage('AI Test Generation') {
            steps {
                // 差分未カバー箇所をCLIエージェントへ渡す
                // 生成されたテストをブランチへコミット
            }
        }
        stage('Run Generated Tests') {
            steps {
                // 全テスト実行（既存＋生成分）
                // 結果をMRコメントへ投稿
            }
        }
    }
}
```

## 6. CLIエージェント（Claude）連携仕様

### 6.1 呼び出し方式

- Jenkins・Webhook受信サーバからシェルコマンドで呼び出す
- 標準入出力でプロンプトを受け渡す

```bash
claude --prompt "$(cat prompt.txt)" --context "$(git diff origin/main...HEAD)"
```

### 6.2 プロンプト設計

#### AIレビュー用プロンプト（概要）

```
以下のdiffを解析し、コード品質・バグ・セキュリティ上の問題を指摘してください。
各指摘には以下を含めてください：
- 指摘ID（R1, R2, ...）
- 対象ファイル・行番号
- 指摘内容
- 修正提案（before/after形式）

出力形式：JSON
```

#### 修正案生成用プロンプト（概要）

```
指摘ID: {ID}
指摘内容: {description}
対象コード: {before}

上記指摘に対する修正後のコードを生成してください。
出力形式：unified diff
```

#### ユニットテスト生成用プロンプト（概要）

```
以下のFlutter/Dartコードの修正内容に対して、
① バグ再現テスト（flutter_testを使用、修正前に失敗するケース）
② 正常・境界テスト（分岐網羅・エッジケース）
を生成してください。
テスト種別：Unit Test（必須）・ Widget Test（UI変更時）
カバレッジ未達箇所: {uncovered_lines}
修正内容: {diff}
```

### 6.3 出力形式（AIレビュー）

```json
{
  "findings": [
    {
      "id": "R1",
      "file": "src/foo.py",
      "line_start": 42,
      "line_end": 45,
      "description": "NullPointerExceptionが発生しうる",
      "suggestion": {
        "before": "return obj.value",
        "after": "return obj.value if obj else None"
      }
    }
  ]
}
```

## 7. GitLab連携仕様

### 7.1 使用API

| 操作 | API |
|---|---|
| MRコメント投稿 | `POST /projects/:id/merge_requests/:mr_iid/notes` |
| MRコメント取得 | `GET /projects/:id/merge_requests/:mr_iid/notes` |
| コメントClose状態取得 | `GET /projects/:id/merge_requests/:mr_iid/discussions` |
| ブランチへのコミット | `POST /projects/:id/repository/commits`（または git push） |

### 7.2 コメント投稿フォーマット（AIレビュー結果）

````markdown
## AI レビュー結果

### R1 ― NullPointerExceptionが発生しうる
**ファイル：** `src/foo.py` L42-45

**指摘：** `obj` が None の場合に例外が発生します。

**修正提案：**
```diff
- return obj.value
+ return obj.value if obj else None
```

> このコメントをCloseすることで指摘を有効確認（CONFIRMED）します。
````

## 8. セキュリティ設計

| 項目 | 対策 |
|---|---|
| Webhook認証 | Secret Tokenによる署名検証（X-Gitlab-Token） |
| AIへの入力制御 | diff・コードのみを渡し、秘密情報（env等）を除外する |
| Git操作権限 | 専用サービスアカウントで最小権限（対象ブランチのpushのみ） |
| コマンド実行 | CLIエージェントはサンドボックス環境で実行（コンテナ分離推奨） |
| DB接続 | ローカル接続のみ許可（外部公開しない） |
