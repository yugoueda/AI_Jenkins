# CI/CDとAIエージェントを統合した自動コードレビュー・ユニットテスト生成システム 技術設計書

**バージョン：** 0.1（ドラフト）  
**作成日：** 2026年5月11日

---

## 1. 目的

本書は、AI自動レビュー・ユニットテスト生成システムの技術的な実装方針・構成・インタフェース仕様を定義する。

言語・フレームワーク非依存のコアアーキテクチャと、プロジェクト固有のスキルファイル・設定を分離することで、**多様なプロジェクトへの展開を容易にする設計**とする。

## 2. システム全体構成

### 2.1 コンポーネント一覧

| コンポーネント | 役割 | 備考 |
|---|---|---|
| GitLab | MR管理・Webhook発火・コメントUI | 社内セルフホスト |
| Jenkins | ビルド・テスト実行・CI基盤 | Pipeline（Jenkinsfile）で管理・localhost限定起動 |
| Webhook受信サーバ | GitLabイベントを受信し処理を振り分け | Python / FastAPI で実装 |
| CLIエージェント | レビュー・修正・ユニットテスト生成を実行 | Claude Code（Claude系） |
| 指摘管理DB | 指摘IDと状態を永続化 | SQLite / PostgreSQL |
| Git操作環境 | AIによるブランチへのコミット | gitコマンド / API |
| nginx | Jenkinsへのリバースプロキシ | 社内NICのみListen、Jenkins直接アクセスを遮断 |
| プロジェクトアダプタ | 言語・FW固有の設定・スキル・ポリシーをパッケージ化 | `project.yaml` + `.claude/` スキルファイル（各リポジトリ配置）§15参照 |

### 2.2 最小構成（1台構成）

```
[社内PC] ──http://192.168.x.x:80──▶ [nginx（社内NICのみListen）]
                                              │ proxy_pass
                                              ▼
[GitLab] ──Webhook──▶ [Webhook受信サーバ]  [Jenkins（127.0.0.1:8080）]
                              │                   ↑ localhost経由
                    ┌─────────┴───────────────────┘
                    ▼
             [CLIエージェント]
                    │
                    ▼
             [指摘管理DB]
                    │
                    ▼
             [Git操作環境]
```

### 2.3 ネットワークアクセス制御

| コンポーネント | バインドアドレス | アクセス可能な接続元 |
|---|---|---|
| Jenkins | `127.0.0.1:8080` | サーバ本体のみ（localhostからのみ） |
| Webhook受信サーバ | `0.0.0.0:8000`（社内NIC） | GitLabサーバ（同一社内ネットワーク） |
| nginx | `192.168.x.x:80` | 社内PCのみ（Jenkinsビルド結果閲覧用） |

## 3. データモデル

### 3.1 指摘（Finding）

```sql
CREATE TABLE findings (
    id          TEXT PRIMARY KEY,  -- AI指摘: R1, R2, ...
    mr_id       TEXT NOT NULL,
    source      TEXT NOT NULL,     -- AI
    status      TEXT NOT NULL,     -- OPEN / APPLIED / REJECTED
    author      TEXT,
    file_path   TEXT,
    line_start  INTEGER,
    line_end    INTEGER,
    description TEXT,
    suggestion  TEXT,              -- before/after形式の修正提案（任意）
    fix_patch   TEXT,              -- 指摘IDに紐づくunified diff（任意）
    fix_patch_sha256 TEXT,         -- 提案外変更混入防止用のパッチハッシュ
    created_at  DATETIME,
    updated_at  DATETIME
);
```

修正パッチは `.patch` ファイルとして保存せず、`findings.fix_patch` にunified diff文字列を直接保持する。  
`fix_patch_sha256` は同じレコードに保持し、`/ai approve ID` 時にDB上のpatch本文とハッシュを照合して提案外変更の混入を防ぐ。

### 3.2 状態遷移

GitLab DiscussionsのClose（Resolve）状態はDBで管理しない。  
全指摘のClose状態は `blocking_discussions_resolved` イベント発火時にDiscussions APIで確認する。

```
OPEN
 ├─▶ APPLIED（/ai approve 実行 → 指摘IDに紐づく修正パッチをハッシュ確認してコミット）
 └─▶ REJECTED（/ai reject 実行）
```

> **全指摘Close後のフロー：**  
> GitLab Discussionsが全Resolve（全指摘Close）された時点で再レビューを自動起動し、新たな問題がなければユニットテスト生成フェーズへ進む。

### 3.3 MR状態

複数MRを並行処理するため、MR状態はMR単位で保持する。

| 状態 | 定義 |
|---|---|
| `Opend` | MR作成直後。ビルド/静的解析エラー時も維持 |
| `レビュー中` | AIレビューまたは再レビュー実行中 |
| `レビュー指摘対応中` | AI/人間レビュー指摘への対応待ち |
| `ユニットテスト生成中` | 指摘解消後、ユニットテスト生成・実行中 |
| `テスト確認中` | 生成テスト結果やカバレッジ確認待ち |
| `マージ待ち` | レビュー・テスト完了後、マージ可能 |
| `Closed` | MRがマージまたはクローズ済み |

ビルドエラー時は `Opend` のまま `BUILD_FIX` を実行する。静的解析エラー時は `Opend` のままMRコメントで通知し、`/ai review` でレビュー開始可能とする。

## 4. Webhook受信サーバ

### 4.1 受信イベント一覧

| GitLabイベント | トリガー条件 | 処理 |
|---|---|---|
| `merge_request` / `opened` | MR作成 | Jenkinsビルドジョブ起動 |
| `note` / MRコメント | `/review <内容>` | 人間指摘はGitLab Discussionとして扱いDB保存しない |
| `note` / MRコメント | `/ai apply <ID>` | 修正案差分を生成しMRコメントに提示 |
| `note` / MRコメント | `/ai approve <ID>` | 指摘IDに紐づく修正パッチをハッシュ確認してコミット |
| `note` / MRコメント | `/ai reject <ID>` | 指摘をREJECTEDに遷移 |
| `note` / MRコメント | `/ai test` | ユニットテスト生成を手動トリガー |
| `note` / MRコメント | `/ai review` | 静的解析失敗後などにAIレビューを手動トリガー |
| `merge_request` / `update` | `blocking_discussions_resolved: true` | Discussions APIで全Resolve確認 → 再レビュー実行 → ユニットテスト生成フェーズへ移行 |

> **⚠️ GitLab制約：個別コメントのResolveはWebhookで検知できない。**  
> 全指摘のClose（全Resolve）は `blocking_discussions_resolved` イベントで検知し、再レビューをトリガーする。

### 4.2 コマンドパーサー仕様

```
受信コメント本文を正規表現で解析
  /review (.+)                               # 人間指摘コメント（DB保存なし）
  /ai (apply|approve|reject|test|review)(\s+\w+)?  # AI操作コマンド
```

#### `/review` コメント仕様

```
/review <指摘内容>
[file: <ファイルパス>]     # 任意
[line: <行番号>]           # 任意
```

例：
```
/review 変数名が仕様と一致していない
file: src/order.py
line: 88
```

- DB登録・ID採番は行わない
- Close状態はGitLab Discussionを正とし、`blocking_discussions_resolved` で全Closeを判断する

- コマンド不正・対象IDが存在しない場合はMRにエラーコメントを返す
- 操作権限は GitLab ロール（Maintainer / Developer）で制御

### 4.3 Webhook署名検証

- GitLab Secret Token をHTTPヘッダ `X-Gitlab-Token` で検証
- 不一致時は 401 を返し処理しない

### 4.4 ジョブキュー・エージェント稼働管理

#### 設計方針

CLIエージェントは1プロセスで動作するため、複数のMRから同時にイベントが届いた場合に排他制御が必要。  
Webhook受信時にジョブをDBに登録し、常駐ワーカーが順次処理する。  
**エージェントの稼働状態はインメモリ変数を使わず、DBの `job_queue` テーブルで判定する。**

#### CLIタスク完了の検知

ワーカーは `asyncio.create_subprocess_exec` でCLIを起動し、`proc.communicate()` でプロセス終了を待機する。  
プロセスが終了した時点でCLIタスクの完了を検知し、DBのジョブステータスを更新する。

```python
async def run_agent(prompt: str) -> tuple[int, str]:
    """CLIエージェントを起動し、終了まで待機して結果を返す"""
    proc = await asyncio.create_subprocess_exec(
        "claude", "--prompt", prompt,
        stdout=asyncio.subprocess.PIPE,
        stderr=asyncio.subprocess.PIPE
    )
    stdout, stderr = await proc.communicate()  # プロセス終了まで待機
    return proc.returncode, stdout.decode()    # 終了コードで成否判定
```

#### ジョブキューテーブル（DB）

```sql
CREATE TABLE job_queue (
    job_id      TEXT PRIMARY KEY,   -- UUID
    mr_id       TEXT NOT NULL,
    event_type  TEXT NOT NULL,      -- BUILD_FIX / REVIEW / APPLY / APPROVE / RE_REVIEW / UNIT_TEST_GEN
    payload     TEXT NOT NULL,      -- JSON（Webhookペイロード等）
    status      TEXT NOT NULL,      -- WAITING / PROCESSING / DONE / FAILED
    created_at  DATETIME NOT NULL,
    started_at  DATETIME,
    completed_at DATETIME
);
```

| event_type | 発生タイミング |
|---|---|
| `BUILD_FIX` | ビルド失敗 → ビルドエラーのみを対象に修正提案 |
| `REVIEW` | MR作成後の静的解析成功、または `/ai review` → AIレビュー実行 |
| `APPLY` | `/ai apply <ID>` コマンド受信 → 修正案差分をMRコメントに提示 |
| `APPROVE` | `/ai approve <ID>` コマンド受信 → 保存済み修正パッチをコミット |
| `RE_REVIEW` | 全指摘Close検知後の再レビュー |
| `UNIT_TEST_GEN` | 再レビューで問題なし → ユニットテスト生成 |

#### エージェント稼働状態の判定（DB参照）

インメモリ変数は使わず、DBの `PROCESSING` 件数で判定する。  
プロセス再起動後も正確な状態を復元できる。

```python
async def is_agent_busy() -> bool:
    """PROCESSING 中のジョブが1件でもあれば BUSY と判定"""
    result = db.query(
        "SELECT 1 FROM job_queue WHERE status='PROCESSING' LIMIT 1"
    )
    return len(result) > 0

async def recover_stale_jobs():
    """起動時: PROCESSING のまま残ったジョブを FAILED に戻す"""
    db.execute(
        "UPDATE job_queue SET status='FAILED' WHERE status='PROCESSING'"
    )
```

#### 処理フロー

```
[Webhook受信]
    │
    ▼
ジョブをjob_queueに登録（status: WAITING）
    │
    ▼
is_agent_busy() でDB確認
    │
    ├─ False（IDLE）──▶ 即時ジョブ取得・status: PROCESSING に更新・処理開始
    │
    └─ True（BUSY） ──▶ キューに積んだまま返却（ワーカーが完了後に拾う）

[CLIプロセス終了を proc.communicate() で検知]
    │
    ▼
returncode で成否判定
    ├─ 0     → status: DONE に更新
    └─ 0以外 → 1回だけリトライ。再失敗時は status: FAILED に更新・MRにエラーコメント投稿
    │
    ▼
次の WAITING ジョブを取得
    ├─ あり ──▶ 次ジョブ処理開始
    └─ なし ──▶ 待機（次のWebhook受信まで）
```

#### 同一MR内の順序制御

同一 `mr_id` のジョブが PROCESSING 中は後続を待機させる（レビュー・コミット・再レビューの競合を防止）。

```python
async def can_process(mr_id: str) -> bool:
    """同一MRで処理中のジョブがあれば待機させる"""
    result = db.query(
        "SELECT 1 FROM job_queue WHERE mr_id=? AND status='PROCESSING'",
        (mr_id,)
    )
    return len(result) == 0
```

## 5. Jenkinsパイプライン設計

### 5.1 ビルドパイプライン（MR作成時）

ビルドと静的解析は分離する。ビルド失敗時は静的解析をskipし、ビルドエラーのみを対象に `BUILD_FIX` ジョブを登録する。静的解析失敗時はAIレビューへ進まず、MR状態はOpendのままMRコメントで通知する。静的解析失敗後は `/ai review` でレビューを手動開始できる。

```groovy
pipeline {
    agent any
    stages {
        stage('Checkout') { ... }
        stage('Build') {
            steps {
                // ${BUILD_COMMAND}（project.yaml / 環境変数 BUILD_COMMAND から取得）
                // 失敗時は BUILD_FIX ジョブを登録し、Lint/AIレビューはskip
            }
        }
        stage('Lint') {
            steps {
                // ${LINT_COMMAND}（同上 LINT_COMMAND）
                // 失敗時はMRコメントで通知し、AIレビューはskip
            }
        }
        stage('AI Review') {
            when {
                expression { currentBuild.currentResult == 'SUCCESS' }
            }
            steps {
                // CLIエージェント呼び出し
                // 指摘をDBに保存・GitLabにコメント投稿
            }
        }
    }
}
```

### 5.2 テスト実行パイプライン（ユニットテスト生成後）

生成済みユニットテストを実行し、テスト結果とカバレッジをMRへ投稿するパイプライン。  
テスト実行失敗時は1回だけリトライする。カバレッジ目標未達時の追加生成ループは自動実行しない。
目標値は Jenkins パラメータで上書き可能で、デフォルトは環境変数（§9.2）から取得する。

#### 処理フロー

```
Checkout
  │
  ▼
既存テスト実行 + lcov カバレッジ取得
  │
  ├─ テスト実行失敗 ──▶ 1回リトライ ──▶ 再失敗時はMRへ投稿（完了）
  │
  ▼
C0 / C1 を LCOV から計算
  │
  ├─ 目標達成 ──────────────────────────────▶ 結果をMRへ投稿（完了）
  │
  └─ 目標未達 ──────────────────────────────▶ 未達箇所と結果をMRへ投稿（完了）
```

> **バグ検出時：** テストは正常に実行されたが生成テストでバグを検出した場合は、修正案提示フローへ戻る。

#### Jenkinsfile

```groovy
pipeline {
    agent any
    parameters {
        string(name: 'MR_IID',    defaultValue: '',
               description: 'GitLab MR IID（Webhook受信サーバから渡す）')
        string(name: 'C0_TARGET', defaultValue: "${env.COVERAGE_C0_TARGET ?: '80'}",
               description: 'C0（ステートメント）カバレッジ目標 (%)')
        string(name: 'C1_TARGET', defaultValue: "${env.COVERAGE_C1_TARGET ?: '70'}",
               description: 'C1（ブランチ）カバレッジ目標 (%)')
    }
    stages {
        stage('Checkout') {
            steps { checkout scm }
        }
        stage('テスト実行・カバレッジ計測') {
            steps {
                script {
                    def c0Target = params.C0_TARGET.toInteger()
                    def c1Target = params.C1_TARGET.toInteger()

                    try {
                        sh "${env.TEST_COMMAND}"  // 環境変数 TEST_COMMAND（例: flutter test --coverage）
                    } catch (err) {
                        echo "テスト実行失敗 → 1回リトライ"
                        try {
                            sh "${env.TEST_COMMAND}"
                        } catch (retryErr) {
                            env.TEST_EXEC_FAILED = 'true'
                            currentBuild.result = 'UNSTABLE'
                        }
                    }

                    if (env.TEST_EXEC_FAILED != 'true') {
                        def finalCov = parseLcov(env.COVERAGE_REPORT_PATH ?: 'coverage/lcov.info')
                        env.FINAL_C0 = finalCov.c0.toString()
                        env.FINAL_C1 = finalCov.c1.toString()
                        env.COVERAGE_MET = (finalCov.c0 >= c0Target && finalCov.c1 >= c1Target).toString()
                    }
                }
            }
        }
        stage('結果をMRへ投稿') {
            steps {
                sh """
                    python src/gitlab_reporter.py \\
                      --mr-id   ${params.MR_IID} \\
                      --c0      ${env.FINAL_C0} \\
                      --c1      ${env.FINAL_C1} \\
                      --c0-target ${params.C0_TARGET} \\
                      --c1-target ${params.C1_TARGET} \\
                      --met     ${env.COVERAGE_MET} \\
                      --test-exec-failed ${env.TEST_EXEC_FAILED ?: 'false'}
                """
            }
        }
    }
}

// ─── 共通ライブラリ関数 ────────────────────────────────────────────────────────

/** LCOV ファイルから C0（ステートメント）と C1（ブランチ）カバレッジを計算する */
def parseLcov(String lcovPath) {
    def lines = readFile(lcovPath).readLines()
    long totalLines = 0, hitLines = 0
    long totalBranches = 0, hitBranches = 0

    lines.each { line ->
        if (line.startsWith('DA:')) {
            // DA:<行番号>,<実行回数>
            totalLines++
            if (line.tokenize(',')[1].toInteger() > 0) hitLines++
        } else if (line.startsWith('BRDA:')) {
            // BRDA:<行番号>,<ブロック>,<分岐番号>,<実行回数 or '-'>
            totalBranches++
            def count = line.tokenize(',')[3]
            if (count != '-' && count.toInteger() > 0) hitBranches++
        }
    }
    return [
        c0: totalLines     > 0 ? (hitLines     * 100 / totalLines).toInteger()     : 100,
        c1: totalBranches  > 0 ? (hitBranches  * 100 / totalBranches).toInteger()  : 100
    ]
}

/** 目標未達の行・分岐を {"lib/foo.dart": [42, 55]} 形式で抽出する */
def extractUncovered(String lcovPath, int c0Target, int c1Target) {
    def lines = readFile(lcovPath).readLines()
    def result = [:]
    def currentFile = ''

    lines.each { line ->
        if (line.startsWith('SF:')) {
            currentFile = line.substring(3)
            result[currentFile] = []
        } else if (line.startsWith('DA:') && currentFile) {
            def parts = line.tokenize(',')
            if (parts[1].toInteger() == 0) {
                result[currentFile] << parts[0].substring(3).toInteger()
            }
        }
    }
    return result.findAll { k, v -> !v.isEmpty() }
}
```

#### カバレッジ種別（C0 / C1）

| 種別 | 内容 | LCOV 対応行 | LCOV 出力対応例 |
|---|---|---|---|
| C0（ステートメント） | 全実行可能行のうち1回以上実行された割合 | `DA:` 行 | Flutter / Python(pytest-lcov) / Jest / JaCoCo変換 |
| C1（ブランチ） | 全分岐のうち1回以上実行された割合（if/else・switch 等） | `BRDA:` 行 | Flutter / Jest / JaCoCo変換 |

> **注：** C2（条件カバレッジ）は多くのLCOV出力に含まれないため対象外とする。LCOV形式にエクスポートできるツール（JaCoCo → lcov、pytest-lcov 等）であれば `parseLcov()` はそのまま流用できる。

## 6. CLIエージェント（Claude）連携仕様

### 6.1 呼び出し方式

- Jenkins・Webhook受信サーバからシェルコマンドで呼び出す
- 標準入出力でプロンプトを受け渡す

```bash
claude --prompt "$(cat prompt.txt)" --context "$(git diff origin/main...HEAD)"
```

### 6.2 プロンプト設計

#### AIレビュー用プロンプト（概要）

```
以下のdiffを解析し、コード品質・バグ・セキュリティ上の問題を指摘してください。
各指摘には以下を含めてください：
- 指摘ID（R1, R2, ...）
- 対象ファイル・行番号
- 指摘内容
- 修正提案（before/after形式）

出力形式：JSON
```

#### 修正パッチ出力（概要）

```
指摘ID: {ID}
指摘内容: {description}
対象コード: {before}

レビュー時に上記指摘に対する修正パッチも生成してください。
出力形式：unified diff
```

#### ユニットテスト生成用プロンプト（概要）

```
以下のFlutter/Dartコードの修正内容に対して、
① バグ再現テスト（flutter_testを使用、修正前に失敗するケース）
② 正常・境界テスト（分岐網羅・エッジケース）
を生成してください。
テスト種別：Unit Test（必須）・ Widget Test（UI変更時）
カバレッジ未達箇所: {uncovered_lines}
修正内容: {diff}
```

### 6.3 出力形式（AIレビュー）

```json
{
  "findings": [
    {
      "id": "R1",
      "file": "src/foo.py",
      "line_start": 42,
      "line_end": 45,
      "description": "NullPointerExceptionが発生しうる",
      "suggestion": {
        "before": "return obj.value",
        "after": "return obj.value if obj else None"
      },
      "fix_patch": "--- a/src/foo.py\n+++ b/src/foo.py\n...",
      "fix_patch_sha256": "<patch sha256>"
    }
  ]
}
```

## 7. GitLab連携仕様

#### 11.2.1 AIレビューエージェント（`ai-reviewer.agent.md`）

**役割：** MRのdiffを受け取り、コード品質・バグ・セキュリティ問題を検出してJSON形式で指摘を出力する。

```markdown
---
name: ai-reviewer
description: >
  AIコードレビューを実行する。MRのdiffを受け取り、コード品質・バグ・セキュリティ問題を
  JSON形式の指摘リストとして出力する。レビュー・静的解析・差分チェックが必要な場合に使用。
tools: [read, search]
user-invocable: false
---

あなたはFlutter/Dartコードの専門レビュアーです。
渡されたdiffを解析し、以下の観点で問題を検出してください。

## レビュー観点

1. **コード品質**: 命名規則・関数分割・単一責任の原則・重複コード
2. **バグリスク**: NullPointerException・型不一致・境界値・非同期処理の抜け
3. **セキュリティ**: 入力値バリデーション不足・秘密情報のハードコード・OWASP Top 10
4. **Dart固有**: `late`変数の未初期化リスク・`Future`の未await・`BuildContext`のAsync越え利用

## 制約

- DO NOT コードを実行・変更する
- DO NOT diff以外のファイルを読み込む（contextに渡された範囲のみ参照）
- ONLY 指摘の検出と出力に専念する

## 出力形式

必ずJSON形式で出力すること。指摘がない場合は `"findings": []` を返す。

```json
{

    {
      "id": "R1",
      "file": "<ファイルパス>",
      "line_start": <開始行>,
      "line_end": <終了行>,
      "description": "<指摘内容>",
      "suggestion": {
        "before": "<修正前コード>",
        "after": "<修正後コード>"
      },
      "fix_patch": "<unified diff>",
      "fix_patch_sha256": "<patch sha256>"
    }
  ]
}
```
```

1. 指摘内容が解消される最小限の変更にとどめる

-<修正前コード>

 <変更なし行>
```
```

void main() {

```
```

## 14. トークン使用量の最適化

Claude APIのコスト削減とレート制限（TPM/RPM）への抵触回避を目的とした設計方針を定める。

### 14.1 基本方針

| 方針 | 概要 |
|---|---|
| プロンプトキャッシング | 変化しない部分をキャッシュし、キャッシュヒット率を最大化する |
| 差分のみ渡す | フルファイルを渡さず `git diff` 出力のみをコンテキストとする |
| モデル使い分け | タスクの複雑さに応じてHaiku / Sonnetを選択する |
| 出力フォーマット圧縮 | JSONのみ要求し、説明文・前置きを排除する |

---

### 14.2 プロンプトキャッシング（Prompt Caching）

Claude APIのPrompt Caching機能（`cache_control: {"type": "ephemeral"}`）を活用し、MR間で共通する固定コンテキストをキャッシュする。

#### キャッシュ構造（メッセージブロック順）

```
[Block 1] システムプロンプト + AGENT.md 定義内容    ← cache_control 付与（最長有効）
[Block 2] コーディング規約・チェックリスト（12節相当）← cache_control 付与
[Block 3] MR固有の diff・カバレッジ情報              ← キャッシュなし（毎回変わる）
```

- キャッシュ有効期間：5分（Claude デフォルト）
- 同一MR内の複数コマンド（REVIEW → APPROVE → RE_REVIEW）ではBlock 1・2 のキャッシュヒットが期待できる

#### 実装例（Python SDK）

```python
response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=4096,
    system=[
        {
            "type": "text",
            "text": AGENT_SYSTEM_PROMPT,          # AGENT.md 内容
            "cache_control": {"type": "ephemeral"}
        },
        {
            "type": "text",
            "text": CODING_CONVENTIONS,           # 12節コーディング規約
            "cache_control": {"type": "ephemeral"}
        }
    ],
    messages=[
        {"role": "user", "content": diff_content} # MR固有 diff（キャッシュなし）
    ]
)
```

### 14.3 入力コンテキストの最小化

フルファイルを渡すとトークン数が数倍に膨らむが、差分のみでは変更箇所に関連する既存コードが欠落するリスクがある。

> **⚠️ 差分のみ渡す場合の欠落リスク：**  
> 変更された関数の呼び出し元・クラスのフィールド定義・別ファイルの `override` 先など、  
> 差分に現れない箇所が影響を受けていても、レビュアーには見えない。

これに対し、**タスク種別とファイルサイズを組み合わせたハイブリッド戦略**を採用する。

#### タスク種別によるオーバーライドルール（最優先）

| event_type | コンテキスト戦略 | 理由 |
|---|---|---|
| `UNIT_TEST_GEN` | **`-W` 固定**（≤ 500行）または **フルファイル**（≤ 200行） | バグ再現テストの生成には変更関数の全体像が必須。前後数行では関数シグネチャ・戻り値・ガード節が欠落する |
| `REVIEW` / `RE_REVIEW` / `APPROVE` | 下記ファイルサイズ基準に従う | レビュー・修正パッチ確認は差分周辺があれば判断可能 |

#### ファイルサイズ基準（REVIEW系タスク）

| 変更ファイルの行数 | 戦略 | 理由 |
|---|---|---|
| ≤ 200行 | **フルファイル送付** | ファイルが小さく、全文渡してもトークンコストが低い |
| 200〜500行 | **関数コンテキスト拡張 (`-W`)** | 変更を含む関数全体を展開し、周辺の定義を補う |
| > 500行 | **差分 + `-U20` 拡張コンテキスト** | 関数全体が大きすぎる場合は前後20行に留める |

#### 関数コンテキスト拡張（`-W`）

`-W` / `--function-context` オプションを使用すると、変更ハンクを含む**関数全体**がコンテキストとして展開される。前後3行のみより格段に文脈が補われる。特に `UNIT_TEST_GEN` では、テスト対象の引数・戻り値・例外仕様がすべて見えることが品質に直結する。

```bash
# 変更を含む関数全体を展開（UNIT_TEST_GEN および 200〜500行ファイルのREVIEW系）
git diff -W origin/main...HEAD \
  -- ':(exclude)*.lock' \
  -- ':(exclude)pubspec.lock' \
  -- ':(exclude)*.g.dart' \
  -- ':(exclude)*.freezed.dart'
```

#### ファイルサイズ・タスク種別判定の実装例

```python
def build_diff_context(changed_files: list[str], event_type: str) -> str:
    """タスク種別とファイルサイズに応じてコンテキスト戦略を切り替える"""
    contexts = []
    for file_path in changed_files:
        line_count = count_lines(file_path)

        if event_type == "UNIT_TEST_GEN":
            # テスト生成は関数全体が必須
            if line_count <= 200:
                contexts.append(f"# {file_path} (full)\n" + read_file(file_path))
            elif line_count <= 500:
                contexts.append(run(f"git diff -W origin/main...HEAD -- {file_path}"))
            else:
                # 500行超は -W だと出力が膨大になるためフルファイルを断念し -U30 で代替
                contexts.append(run(f"git diff -U30 origin/main...HEAD -- {file_path}"))
        else:
            # REVIEW / APPROVE 系
            if line_count <= 200:
                contexts.append(f"# {file_path} (full)\n" + read_file(file_path))
            elif line_count <= 500:
                contexts.append(run(f"git diff -W origin/main...HEAD -- {file_path}"))
            else:
                contexts.append(run(f"git diff -U20 origin/main...HEAD -- {file_path}"))

    return "\n\n".join(contexts)
```

#### 除外対象ファイル（共通）

```bash
# 自動生成ファイルはいずれの戦略でも除外する
EXCLUDE_PATTERNS=(
  ':(exclude)*.lock'
  ':(exclude)pubspec.lock'
  ':(exclude)*.g.dart'
  ':(exclude)*.freezed.dart'
)
```

| 除外対象 | 理由 |
|---|---|
| `pubspec.lock` | 自動生成・依存解決ファイル（レビュー対象外） |
| `*.g.dart` / `*.freezed.dart` | コード生成ファイル（手動編集なし） |
| `*.lock` 全般 | パッケージロックファイル |

### 14.4 モデル選択（タスク別）

タスクの複雑さに応じてモデルを使い分け、不必要に高性能なモデルを使用しない。

| event_type | 推奨モデル | 根拠 |
|---|---|---|
| `REVIEW` | `claude-sonnet-4-5` | セキュリティ解析・複合的判断が必要 |
| `APPROVE` | `claude-sonnet-4-5` | 保存済みパッチの整合性確認が必要 |
| `RE_REVIEW` | `claude-sonnet-4-5` | 再レビューもフルレビューと同等 |
| `UNIT_TEST_GEN` | `claude-sonnet-4-5` | テンプレートベースだが品質重視 |
| エラーコメント生成 | `claude-haiku-4-5` | 定型文生成のみ・複雑な判断不要 |

CLIオプションでモデルを指定する。

```bash
# レビュー・修正系
claude --model claude-sonnet-4-5 --prompt "$(cat prompt.txt)"

# 定型コメント生成等の軽量タスク
claude --model claude-haiku-4-5 --prompt "$(cat error_comment_prompt.txt)"
```

### 14.6 トークン使用量のモニタリング

APIレスポンスの `usage` フィールドをジョブ完了時にDBへ記録し、コスト管理・最適化の継続改善に活用する。

#### job_queue テーブルへの追加カラム

```sql
ALTER TABLE job_queue ADD COLUMN input_tokens        INTEGER;
ALTER TABLE job_queue ADD COLUMN output_tokens       INTEGER;
ALTER TABLE job_queue ADD COLUMN cache_read_tokens   INTEGER;  -- キャッシュヒット分（割引対象）
ALTER TABLE job_queue ADD COLUMN cache_write_tokens  INTEGER;  -- キャッシュ書き込み分
```

#### 記録実装例

```python
async def run_agent(prompt: str, model: str) -> tuple[int, str, dict]:
    proc = await asyncio.create_subprocess_exec(
        "claude", "--model", model,
               "--output-format", "json",   # 使用量をJSON出力に含める
               "--prompt", prompt,
        stdout=asyncio.subprocess.PIPE,
        stderr=asyncio.subprocess.PIPE
    )
    stdout, stderr = await proc.communicate()
    result = json.loads(stdout.decode())

    usage = {
        "input_tokens":       result.get("usage", {}).get("input_tokens", 0),
        "output_tokens":      result.get("usage", {}).get("output_tokens", 0),
        "cache_read_tokens":  result.get("usage", {}).get("cache_read_input_tokens", 0),
        "cache_write_tokens": result.get("usage", {}).get("cache_creation_input_tokens", 0),
    }
    return proc.returncode, result.get("content", ""), usage
```

## 15. プロジェクトアダプタ設計

多様なプロジェクトへ導入できるよう、**コア（言語非依存）** と **アダプタ（言語・FW固有）** を分離する。

### 15.1 設計方針

| 層 | 内容 | 変更頻度 |

| アダプタ | ビルド/Lint/テストコマンド・レビュー観点・テストテンプレート | 導入先ごとに差し替え |

### 15.2 アダプタ構成（対象リポジトリ側）

導入先リポジトリのルートに `.ai-review/` ディレクトリを配置する。

```
<導入先リポジトリ>/
├── .ai-review/
│   ├── project.yaml           # プロジェクト設定（必須）
│   └── skills/                # 言語固有スキル（§11 の .claude/skills/ 相当）
│       ├── code-review/
│       │   ├── SKILL.md       # レビュー観点・チェックリスト
│       │   └── checklists/
│       └── unit-test/
│           ├── SKILL.md       # テスト生成規約・テンプレート
│           └── templates/
└── ...（通常のプロジェクトファイル）
```

### 15.3 project.yaml スキーマ

```yaml
# .ai-review/project.yaml
language: flutter              # 必須：言語識別子
build_command: flutter build   # 必須：ビルドコマンド
lint_command: flutter analyze  # 必須：Lint コマンド
test_command: flutter test --coverage   # 必須：テスト実行コマンド（カバレッジ出力込み）
coverage_format: lcov          # 任意：デフォルト lcov（現在 lcov のみサポート）
coverage_report_path: coverage/lcov.info  # 任意：レポートファイルパス
```

Webhook受信サーバ起動時に `project.yaml` を読み込み、§9.2 の対応する環境変数へ注入する。`.env` の値が未設定の場合のフォールバックとして機能する。

### 15.4 言語別対応状況

| 言語 / FW | test_command 例 | カバレッジ形式 | 対応状況 |
|---|---|---|---|
| Flutter (Dart) | `flutter test --coverage` | LCOV | ✅ リファレンス実装 |
| Python | `pytest --cov --cov-report=lcov` | LCOV (pytest-lcov) | 🔧 スキルファイル要追加 |
| TypeScript / Node | `jest --coverage --coverageReporters=lcov` | LCOV | 🔧 スキルファイル要追加 |
| Java / Kotlin | `./gradlew test jacocoTestReport`（lcov変換別途） | LCOV (JaCoCo→lcov) | 🔧 スキルファイル要追加 |

> **重要：** `parseLcov()` 関数（§5.2）はLCOV形式に準拠した出力であれば言語を問わず動作する。各言語のカバレッジツールをLCOV出力に設定するだけでコア側の変更は不要。
