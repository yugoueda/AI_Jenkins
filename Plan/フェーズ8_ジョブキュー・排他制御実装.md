## フェーズ8：ジョブキュー・排他制御実装 作業計画

### 前提条件

- フェーズ1の `job_queue` テーブルが存在すること
- フェーズ3のCLI dispatcherがジョブ単位で呼び出せること
- 稼働状態はインメモリではなくDBで判定すること

### 作成ファイル一覧

```
src/queue/

```

### アイテム別設計

#### 8-1 ジョブ登録機能

- Webhook受信時に `WAITING` で `job_queue` へ登録
- `event_type` は `REVIEW` / `APPROVE` / `RE_REVIEW` / `UNIT_TEST_GEN`
- payloadはJSON文字列で保存

#### 8-2 常駐ワーカー

- `WAITING` ジョブを作成順に取得
- 処理開始時に `PROCESSING`、成功時に `DONE`、失敗時に `FAILED` へ更新
- CLI終了は `proc.communicate()` で検知

#### 8-3 エージェント稼働状態判定

- `PROCESSING` ジョブが1件以上あればbusy
- Webhook応答はキュー登録完了時点で返す

#### 8-4 同一MR内順序制御

- 同一 `mr_id` の `PROCESSING` がある場合、後続ジョブは待機
- `APPROVE` → `RE_REVIEW` の順序依存を保護

#### 8-5 staleジョブ復旧

- 起動時に `PROCESSING` のまま残ったジョブを `FAILED` に更新
- 必要に応じて失敗理由へ `recovered as stale` を記録
